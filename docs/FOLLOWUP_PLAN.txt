  2026-07-18: Fixed a real crash-on-cancel bug reported by the user
  ("program crashes when I cancel rendering"), root-caused via WER minidump
  analysis rather than guesswork - the code review alone looked correct
  (RenderDialog/RenderAllDialog/FullscreenView all set cancelRequested and
  join their worker QThread cleanly on teardown; RenderWorker::renderFull
  already unconditionally writes the output PNG on a cancelled render, so
  "save the partial render on cancel" was already implemented). Windows'
  local crash-dump collection (%LOCALAPPDATA%\CrashDumps) had several real
  apo_gui.exe .dmp files from the user's own sessions; no debugger was
  installed (no cdb/windbg on this machine), so a small standalone minidump
  reader/symbolizer was written on the spot (C#, P/Invoke into dbghelp.dll -
  parses the MINIDUMP_HEADER/stream directory by hand, loads every module it
  can still find on disk at its recorded base address via SymLoadModuleExW,
  and both runs a proper StackWalk64 and does a raw stack-memory scan for
  candidate return addresses as a fallback when precise unwinding through
  system DLLs isn't available). The most recent dump's exception was
  0xC0000409 at ucrtbase!abort - not memory corruption despite the
  "STATUS_STACK_BUFFER_OVERRUN" name (Windows reuses that code for CRT
  fastfail); the raw stack scan showed ucrtbase!terminate, VCRUNTIME140's
  exception-dispatch frames (_CxxFrameHandler4/_processing_throw), right
  next to thread-entry-trampoline frames - the signature of an uncaught C++
  exception unwinding out of a worker thread's top-level function, which
  the standard mandates calls std::terminate() immediately and uncatchably
  by anything on a different thread's stack. Nothing in the render path
  (RenderWorker's three slots, or the std::jthread pool Renderer::render()
  spawns for the multi-threaded case) had a try/catch anywhere, so any
  exception there - most plausibly std::bad_alloc, since a user is most
  likely to reach for Cancel on exactly the large/slow/memory-heavy renders
  where an allocation is most likely to fail - took the whole process down
  instead of just failing that one render. Fixed by wrapping all three
  RenderWorker slots (renderFlame/renderFlameWithProgress/renderFull) in
  try/catch, converting any exception into a normal "finished, not saved"
  signal instead of a crash; and, separately, by making the multi-threaded
  jthread pool in Renderer.cpp catch per-worker exceptions into a shared
  std::exception_ptr and rethrow on the calling thread after every worker
  joins - necessary because a jthread's own uncaught exception terminates
  immediately regardless of any try/catch elsewhere, so RenderWorker's own
  guard alone would not have covered that path. Could not get a byte-exact
  reproduction in the offscreen test harness (30 iterations of
  render-then-cancel-mid-flight on a 1600x1200/density-4000 render, timed to
  land after real progress had ticked, all passed cleanly - this machine
  has ~25GB free RAM, so a memory-pressure trigger needs either a much
  larger request or a tighter machine to hit reliably), so this is a
  defensive-correctness fix verified via full rebuild + full 43/43 ctest
  pass in Debug, not a fix confirmed against a reproduced local crash.

  src/ui/MemoryBudget.h/.cpp (UI layer, not core - GlobalMemoryStatusEx is a

  the clamping arithmetic against a known figure instead of whatever's

persist the library as a whole. This quietly breaks the port's own stated
contract: MainWindow.h's Rename/Delete doc comment says "a renamed or
pruned library still needs Save Flame As to persist anywhere" - but Save
Flame As can't do that. A user who renames/reorders/deletes/duplicates
their way to a curated library cannot save it.
Plan: add "Save All As..." next to the existing action; apo::saveFlameFile
already accepts a vector of flames, so this is UI wiring plus a test that
round-trips a multi-flame library through save/load. Cheap, closes a real
data-loss-shaped hole.

================================================================================
PART B - PERFORMANCE OPPORTUNITIES
================================================================================
Context: the renderer's architecture is already good - per-thread private
bucket arrays (zero contention), xoshiro256** RNG, propTable xform
selection, precomputed color/tone/filter tables, batch-granular
cancellation. The headroom is in dispatch overhead per point, setup cost
per render, memory traffic, and two leftover single-threaded passes.

IMPORTANT GLOBAL CAVEAT: several items below change the RNG draw sequence
or floating-point rounding, which changes rendered output bit-for-bit.
baseline_regression_test.cpp's checked-in reference images would need
regenerating in the same change that lands any such item - do them
deliberately, one at a time, with before/after images eyeballed.

--------------------------------------------------------------------------------
B0. FIRST: a points-per-second benchmark harness           [prerequisite]
--------------------------------------------------------------------------------
None of the wins below should be landed on faith. Add a small benchmark
target (not a ctest - a manually-run tool, or a tests/ binary excluded
from the default suite) that renders 2-3 representative flames
(linear-only, transcendental-heavy e.g. julian/swirl, blur-heavy) at fixed
seed/size/density, single- and multi-threaded, and reports points/sec and
wall time per phase (setup / iterate / reduce / tonemap). Every item below
then gets a measured before/after in this file's progress log.

--------------------------------------------------------------------------------
B1. Render setup cost: Flame construction/clone is enormously
    over-allocating                                    [BIG for interactivity]
--------------------------------------------------------------------------------
Verified: Flame's constructor allocates 101 XForms (kMaxXForms=100 + final,
Flame.h:110). EVERY XForm constructor instantiates one object for EVERY
registered variation (~150-200: 94 builtin variation files + 48 plugin
wrappers, XForm.cpp addRegVariations) plus a std::function list. So one
Flame() is ~15,000-20,000 heap allocations - and Renderer::render() calls
flame.clone() ONCE PER THREAD (Renderer.cpp:727), so a 16-thread render
does ~300,000 allocations before the first point is iterated. Worse,
XForm::assign() (used by clone) copies every variation's variables BY NAME
through virtual getVariable/setVariable string comparisons - hundreds of
thousands of string ops per clone wave. This is invisible on a 5-minute
final render but is a real, recurring cost for every interactive preview
render (camera drags, editor drags, thumbnails, mutate previews).
Plan, in increasing order of invasiveness (measure after each):
  (a) Clone only the active xforms for rendering: a
      Flame::cloneForRender() (or a leaner internal path in render()) that
      copies numXForms()+final xforms instead of all 101. ~10-50x less
      clone work for typical 2-10-xform flames. No behavior change.
  (b) Lazy variation instantiation per XForm: only variations with nonzero
      weight (plus those touched via getVariable/setVariable) need live
      objects; name lookups for dormant ones can consult
      VariationRegistry's factories (VariationFactory already exposes
      numVariables/variableNameAt without an instance - Variation.h:89).
      Largest structural win; touches XForm's internals broadly, so it
      needs the full suite + baselines green after.
  (c) Index-based variable copy in XForm::assign()/
      interpolateVariablesFrom() (getVariableAt/setVariableAt(int)) to
      kill the per-copy string traffic that remains after (b).

--------------------------------------------------------------------------------
B2. Hot-loop dispatch: std::function chain + virtual RNG    [BIG, measure]
--------------------------------------------------------------------------------
Two indirect-dispatch costs sit inside the per-point loop:
  (1) XForm::nextPoint runs `for (auto& fn : calcFunctionList_) fn();`
      where each entry is a std::function<void()> (XForm.cpp:221) -
      type-erased indirect call per active variation per point, opaque to
      the inliner.
  (2) Rng::uniform01() is a VIRTUAL call (Rng.h:39), and it fires several
      times per point: the propTable draw, the opacity roll, stochastic
      variations, DOF draws. ThreadState holds unique_ptr<Rng>, so the
      compiler cannot devirtualize any of it.
Plan:
  (a) RNG: make the render path use the concrete Xoshiro256Rng type
      (ThreadState holds Xoshiro256Rng by value; renderBatch takes it
      concretely). XForm's stochastic variations still take Rng& - that
      boundary can stay virtual initially (measure), or XForm can be
      templated/overloaded on the concrete type later if the benchmark
      says it matters.
  (b) calcFunctionList_: replace std::function entries with a flat array
      of {void(*fn)(void*), void* ctx} pairs or member-function-pointer +
      object pairs (both allocation-free and predictably branch-target-
      cached). The prepare()-time flexibility (julian's
      selectCalcFunction) is preservable - selectCalcFunction can return
      an enum/function-pointer instead of a std::function.
Expected: single-digit to low-double-digit % on transcendental-light
flames (where dispatch is a bigger fraction of per-point cost); less on
sin/cos-heavy ones. Benchmark decides how far to push.

--------------------------------------------------------------------------------
B3. Skip the opacity roll when transOpacity == 1        [SMALL, cheap]
--------------------------------------------------------------------------------
renderBatch pays `rng.uniform01() >= xf.transOpacity` per point
(Renderer.cpp:507) even though transOpacity is 1.0 for virtually every
xform in real flames (the roll can then never reject). Precompute a
per-xform "always passes" flag at prepare time and branch on it - saves
one RNG draw + compare per point in the common case.
CAVEAT: removes draws from the RNG stream -> single-threaded deterministic
output changes -> baselines must be regenerated in the same change.

--------------------------------------------------------------------------------
B4. Parallelize the two leftover single-threaded passes    [MEDIUM, easy]
--------------------------------------------------------------------------------
Both run after iteration, both trivially data-parallel, both currently
sequential:
  (a) Bucket reduction (Renderer.cpp:793-805): threads x bucketArea x 32B
      of streaming adds. For a 3200x2400 os=2 render that's ~50M buckets;
      16 source arrays = ~25GB of memory traffic through one core. Split
      the bucket index range across the same worker threads (each thread
      sums ALL sources for ITS slice - no synchronization needed).
      The comment at :786 calls this negligible; at preview sizes it is,
      at large multi-thread renders it is measurably not.
  (b) toneMap(): per-pixel filter convolution + pow()-heavy
      gamma/vibrancy, plus (with adaptive DE on) a per-pixel variable-
      radius kernel pass - all one thread. Parallelize over output rows
      (each row's writes are disjoint; all bucket reads are read-only).
      With DE enabled at large sizes this pass alone can be seconds.
Neither changes output at all (pure reordering of independent work) -
no baseline impact, easy tests (same image bytes as single-threaded).

--------------------------------------------------------------------------------
B5. Bucket memory: 32-byte doubles -> optional 16-byte floats   [EXPERIMENT]
--------------------------------------------------------------------------------
Bucket = 4 doubles (Renderer.cpp:30). The chaos game's bucket writes are
random-access over a huge array - this is the classic flame-fractal
bottleneck (cache misses, not arithmetic). Halving the bucket to 4 floats
(or float RGB + uint32 count) halves working-set size and write traffic,
AND halves the per-thread-copy memory bill (see B6).
CAVEATS to evaluate honestly: float count saturates addition at ~16.7M
points/bucket (high-density renders concentrate exactly like that at
attractor hot spots); accumulated float RGB rounds differently ->
baselines change. Suggested shape: template the accumulation/reduction on
the bucket scalar; previews (low density, latency-sensitive) use float,
RenderDialog finals keep double - or make it an Options setting after
measuring whether the quality delta is even visible.

--------------------------------------------------------------------------------
B6. Memory-budget-aware thread count                    [SAFETY/PERF UX]
--------------------------------------------------------------------------------
Peak memory = (threads+1) x bucketArea x 32B (estimatePeakMemoryBytes).
At 3200x2400 with oversample 3, one copy is ~2.2GB - 16 threads + 1 wants
~37GB and will page-thrash (a performance catastrophe far worse than
running fewer threads). The dialog SHOWS the estimate but nothing acts on
it. Plan: before launching, clamp effective thread count so the estimate
fits a budget (e.g. a fraction of physical RAM via
GlobalMemoryStatusEx, or an AppSettings cap), status-line note when
clamped ("using 6 threads to stay under N GB"). Pairs naturally with B5
(floats double the affordable thread count).

--------------------------------------------------------------------------------
B7. Compiler flags for apo_core                          [CHEAP, measure]
--------------------------------------------------------------------------------
CMakeLists.txt sets only /W4 /permissive- /EHsc; Release gets MSVC's
default /O2. Not set: /arch:AVX2 (or /arch:AVX512) - worth benchmarking
for the math-heavy core (opt-in CMake option, since it drops pre-2013
CPUs).
WARNING - do NOT reach for /fp:fast: renderBatch's poisoned-point bailout
(Renderer.cpp:496) and the accept-path check (:516) depend on NaN/Inf
propagating honestly through variation math into std::isfinite. /fp:fast
licenses the compiler to assume no NaNs and can silently break exactly
that logic. If floating-point relaxation is ever explored, it must be
file-scoped away from the isfinite-dependent code and A/B'd against
baselines; default recommendation is to leave /fp precise.

--------------------------------------------------------------------------------
B8. Preview-density cap for the library's full-quality render   [UX CHOICE]
--------------------------------------------------------------------------------
MainWindow renders each selected flame at its own full width/height/
sampleDensity (only camera-drag previews are reduced). A library of
density-2000 flames means every arrow-key selection change launches a
multi-second render. Options if this ever bothers in practice: cap the
selection render's density at some multiple of
AppSettings::previewSampleDensity(), or add an Options "preview quality"
setting. Flagged, not recommended by default - the current behavior is
the deliberate P2.5-era choice and the coalescing gate already prevents
queue pileups.
