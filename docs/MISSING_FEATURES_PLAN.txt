  2026-07-17  P4.1 + P4.3 DONE - the last two items on the plan's own
              suggested order of work.
              P4.1 (memory-usage estimate): a new
              Renderer::estimatePeakMemoryBytes(flame, threadCount) static
              method (Renderer.h/.cpp), deliberately implemented by
              *reusing* the existing buildFilter/buildBuffersAndCamera
              internal functions render() itself already calls (same call
              order: buildBuffersAndCamera reads plan.filterSize, set by
              buildFilter) rather than hand-copying their bucket-sizing
              formulas into the UI layer, where they could silently drift
              out of sync as Renderer.cpp's own math evolves. Estimate is
              (threadCount+1) * bucketWidth * bucketHeight * sizeof(Bucket)
              - one private bucket array per render thread (the lock-free-
              per-thread design Renderer.h's own class comment already
              documents) plus one more for the final reduction buffer every
              thread's buckets get summed into; every other allocation
              (filter kernel, per-thread cloned Flame, output pixels,
              tone-map tables) is negligible by comparison and omitted.
              Pure arithmetic, no chaos-game iteration - cheap enough to
              recompute on every RenderDialog spin-box edit. RenderDialog
              gained a live "Estimated Peak Memory" label, recomputed on
              width/height/oversample/filter-radius changes only (sample
              density deliberately excluded - it changes point count, not
              bucket array size, confirmed via a dedicated test that
              raising density leaves the label unchanged).
              P4.3 (Save parameters): a new "Save parameters (.flame)
              alongside the image" checkbox; on a successful (uncancelled,
              successfully-saved) render, writes a .flame with the same
              basename as the PNG next to it, capturing the *exact*
              settings that were actually rendered (the dialog's own
              width/height/density/oversample/filter-radius/adaptive-filter
              at Render time - a new pendingRenderFlame_ member captures
              the real clone startRender() launched, since flame_ itself
              may differ from what was actually rendered).
              Tests: 3 new checks in renderer_test.cpp

  2026-07-17  P2.5 DONE (by request) - elapsed/remaining time in the
              status bar during the main preview's full-quality render.
              Verified the original's actual mechanism before porting it:
              Main.pas's OnProgress callback (Renderer.OnProgress) updates
              a progress bar plus separate status-bar panels for elapsed
              and estimated-remaining time, using
              `remaining := elapsed/prog - elapsed` (extrapolate a total
              from progress-so-far, subtract what's already elapsed) -
              ported that exact formula, not a guessed one.
              Scope note, decided deliberately rather than by omission:
              P2.5's plan text bundles two distinct things under
              "progressive main preview" - (a) live elapsed/remaining/
              percent status, and (b) the bitmap itself visibly refining
              in place while the render is in flight. Only (a) is
              implemented. (b) is a real architectural difference, not a
              missing nicety: the original's incremental appearance comes
              from its render thread writing directly into a shared
              framebuffer the preview control blits from, tightly coupled
              to its one-shared-buckets-array-plus-lock threading model;
              this port's Renderer::render() is a deliberately-chosen
              one-shot blocking call with no intermediate/streamable state
              (see Renderer.h's own class comment on trading that
              capability away for zero-contention per-thread private
              bucket arrays). Reproducing (b) would mean building a
              second, fundamentally different incremental render path
              alongside the existing one - a large, real undertaking, not
              proportionate to an item the plan itself flagged as
              lowest-priority/optional.
              Implementation: RenderWorker gained a new
              renderFlameWithProgress() slot (NOT an added parameter on
              the existing renderFlame() - confirmed the hard way, via a
              failed build, that Qt's connect() rejects a slot with more
              parameters than its signal even when the extra one has a
              default value, since a default doesn't change the slot's
              actual function-pointer type; widening renderFlame() itself
              would have broken all 5 other windows/dialogs already
              connected to it). MainWindow's own renderRequested signal
              gained a RenderProgress* parameter (a class-local change -
              every other window's identically-named-but-distinct
              renderRequested signal is untouched); only its full-quality
              preview-render path constructs and passes one, matching
              plan's original scope of "the main preview" specifically -
              the cheap reduced-quality render P2.1's camera dragging
              fires on every mouse-move deliberately does not get a
              progress readout, both because those finish fast enough not
              to need one and because flickering percent text on every
              drag tick would be a UX regression, not an improvement.
              Tests: 1 new end-to-end check in
              main_window_interaction_test.cpp (a real, sized-to-take-a-
              perceptible-but-bounded amount of time render shows a live

HEADLINE FINDING

The Editor screenshot (093205.png) exposes the single biggest gap in the
port: the entire right-hand transform-properties panel is missing. Verified
against the code: apophysis-cpp has NO UI whatsoever for editing variation
weights, variation variables (named parameters), transform weight (density),
per-transform color/color-speed/opacity, per-transform names, or the final
transform. The port can drag triangles, add/delete/duplicate transforms,
edit xaos, and randomize via Mutate - but it cannot actually AUTHOR a flame:
there is no way to say "make this transform a spherical" or "give this
transform weight 0.3". This is the core creative workflow of the original
application and must be Priority 1.

  1. "Gradient source on random flame" (Options > Random > "On random flame

     at RndFlame.pas:185 (`case randGradient of`) and Main.pas:932/1073.
     The earlier grep searched the UI control name, not the backing
     variable. This setting is live and now planned (P3.2).
  2. The Variations enable/disable checklist (Options > Variations) IS

PRIORITY 1 - EDITOR TRANSFORM-PROPERTIES PANEL (the app's core feature)
--------------------------------------------------------------------------------
Reference: screenshot 093205.png; original Forms/Editor.pas right panel.

Replace/extend EditorWindow's bare xform QListWidget with a proper side
panel containing (top to bottom, as in the original):

1.1  Transform selector + header row
     - Transform combo (all transforms + "Final" when enabled), Name text
       field (XForm::transformName - already in core, round-trips through
       FlameIO), Weight numeric edit (XForm::density).
     - CAUTION on weight edits: Flame::numXForms() treats density==0 as the
       end-of-list sentinel, so the weight editor must clamp to a small
       positive minimum (the original's own UI forbids 0 too).

1.2  Variations tab
     - Table of every variation name (local + registered; the port already
       has VariationRegistry with the full set) with editable weight values.
     - Search/filter box, "Hide unused variations" checkbox, Clear button -
       all visible in the screenshot and cheap to build on QSortFilterProxy.
     - Core support already exists: XForm::numVariations()/variation()/
       setVariation(). This tab is pure UI wiring.

1.3  Variables tab
     - Name/value table of the selected transform's registered-variation
       parameters (julian_power etc.). Core support exists:
       XForm::getVariable()/setVariable() + VariationRegistry's per-type
       variable-name lists. Show only variables of variations with nonzero
       weight (matching the original's behavior).

1.4  Colors tab
     - Color position slider (XForm::color, 0..1) with gradient-strip
       preview showing the picked color; Color speed (XForm::symmetry,
       -1..1); Opacity (XForm::transOpacity, 0..1); Direct-color
       (XForm::pluginColor). All fields exist in core + FlameIO; no UI.
     - Include the "solo transform" toggle here (Flame::soloXform - the
       renderer already honors it; no UI exists).

1.5  Transform tab (numeric affine editing)
     - Editable a/b/c/d/e/f coefficient fields for the selected transform's
       main (c) and post (p) matrices, live-synced with the triangle canvas
       drags. The original shows these as X/Y/O vector pairs; either layout
       is fine, values must round-trip with the canvas.

1.6  Triangle tab
     - Per-vertex coordinate editing plus the exact-value transform tools
       from the original (rotate by N degrees, scale by N percent, and the
       toolbar's rotate-90-CW/CCW + flip-horizontal/vertical buttons).
       XForm::rotate()/scale()/multiply()/translate() already exist in core.

1.7  Final transform support in the UI
     - The core (Flame::finalXform/finalXformEnabled/useFinalXform) and the
       renderer fully support final transforms, and FlameIO round-trips
       them - but TriangleCanvas never draws the final xform and
       EditorWindow has no way to create, toggle, select, or edit one. Add:
       "Enable final transform" toolbar toggle, canvas drawing/selection of
       the final triangle (the original draws it as a distinct color), and
       inclusion in the transform selector.

1.8  Copy/paste transform
     - Original toolbar has copy/paste of a whole transform (matrix +
       variations + colors) via clipboard. Implement as in-app copy/paste
       (a stored XForm snapshot, pasted via XForm::assign) - the original's
       text-clipboard interchange format is optional.

     All of 1.1-1.8 must integrate with EditorWindow's existing undo stack.
     The Structural undo entry kind (whole-flame snapshots) already added
     for Add/Delete/Duplicate can capture every one of these edits without
     new undo machinery; batch continuous slider drags into one entry via
     the existing begin/commit gesture pattern used by AdjustDialog.

     Estimated size: the largest single work item in the port so far -
     comparable to Curves + Xaos + PostProcess combined. Suggest splitting
     into three deliverables: (a) 1.1+1.2 variations+weight (unblocks
     authoring), (b) 1.3+1.4 colors+variables, (c) 1.5-1.8 numeric/final/
     copy-paste.
